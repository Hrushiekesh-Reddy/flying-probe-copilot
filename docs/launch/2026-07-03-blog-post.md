# Building an AI co-pilot for PCBA test analytics

*A queryable spine, SPC that runs itself, and a citation-forced RAG layer — over 8 weeks of evenings and weekends.*

**By Hrushiekesh Reddy Kanjula** · Manufacturing Engineer, ~4 years PCBA · Dallas, TX
**Repo:** [flying-probe-copilot](https://github.com/Hrushiekesh-Reddy/flying-probe-copilot) · **Case study:** [docs/case-study.md](https://github.com/Hrushiekesh-Reddy/flying-probe-copilot/blob/main/docs/case-study.md)

## The gap

Every PCBA line that runs flying-probe or in-circuit testers emits structured logs — millions of measurement records per shift. The data is there. The analytics layer usually isn't. Factories pay for a commercial suite or do it in Excel, and root-cause investigations still come down to "we saw something like this on lot 2024-W18, ask Kevin." That works until Kevin takes a week off.

The four gaps that actually cost time on the floor:

1. **No queryable spine** — logs are flat files; correlating a shift's yield drop against a component family takes hours.
2. **No SPC automation** — Wheeler XmR, run-of-8, zone rules are well-known but rarely automated against PCBA test data.
3. **No anomaly detection** — per-component failure-rate outliers ("Q15 is failing 4× as often as line average") fall through.
4. **No grounded root-cause Q&A** — "why are these 0402s tombstoning?" should be a query returning a cited answer, not a Slack message.

I spent the last two months closing those four gaps for the open-source, local-first, single-engineer case. This post walks through what shipped and the three engineering bugs I learned the most from.

## What's in it

- A **synthetic HP3070 / i3070 log generator** — Keysight Log Record Format, three board profiles, configurable fault correlation, shift-physics-aware timestamps. ~1 second per 1,000 panels.
- A **parser + DuckDB spine** — 9-table schema (5 dim + 1 run + 3 fact), byte-precise round-trip integrity.
- An **analytics library** — four pure Python functions: `yield_over_time`, `failure_pareto`, `individuals_chart` (Wheeler XmR + run-of-8), `z_score_anomalies` (leave-one-out severity-first).
- A **hybrid RAG co-pilot** — ChromaDB-cosine + rank-bm25 with Reciprocal Rank Fusion over an 8-doc failure-mode knowledge base. Forced citations. Refuses when evidence is insufficient.
- A **Streamlit dashboard** — six pages (Overview / Yield / Pareto / SPC / Anomalies / Co-Pilot).

**Verification:** 667 passing tests, 97% line coverage, 10/10 on the live RAG eval in 37 seconds.

## The architecture (in one paragraph)

The analytics pipeline and the RAG pipeline are independent. Logs → parser → DuckDB → four analytics functions → four dashboard pages. Knowledge base → heading-aware chunker → dual index (ChromaDB HNSW + BM25) → hybrid retriever with Reciprocal Rank Fusion → `answer()` with strict anti-hallucination grounding → Streamlit Co-Pilot chat. Gemini 3.5 Flash is the only thing that touches the network, and only when `RAG_RUN_LLM_EVAL=1` in tests or when a real user asks a question. The whole stack runs against a single `.duckdb` file on a laptop.

## Three bugs, three lessons

Every phase had exactly one bug that made me change how I think about a whole class of problems.

### BUG-004 — the shift-snap that lied about time

The synthetic generator schedules panels across three shifts. Shift C runs 22:00 to 05:59 — so a raw timestamp of 02:00 belongs in shift C's window that *started* at 22:00 the **previous** day. My first cut drew a shift letter uniformly per panel and then snapped the timestamp into that shift's window. The shift-C overnight wrap was a literal `pass` statement.

The result: a 02:00 raw draw randomly assigned to shift C would silently jump to 22:00 on the **same** day. SPC trend analysis on that dataset would have shown completely fictitious shift-vs-shift yield deltas.

The fix wasn't to patch the wrap. It was to derive shift from time-of-day, not draw it independently. `_shift_for_hour(hour)` returns the physically correct shift; `_shift_window_start(ts, shift)` snaps to the right window-start (previous 22:00 for early-morning shift C). The broken wrap became unreachable.

**Class of bug:** any time two correlated parameters (here: shift letter + timestamp) are drawn independently and then "reconciled," the reconciliation step is a bug factory. Derive one from the other.

### BUG-013 — the model that got retired mid-evaluation

The Phase 3 exit criterion was a live ≥8/10 evaluation against Gemini on a frozen 10-question dataset. First attempt ran 2 minutes 31 seconds and failed with `404 This model models/gemini-2.0-flash is no longer available`. Google had retired it. The entire 2:31 wall-clock was gRPC's 600-second retry deadline timing out, not real API work.

Three-line fix: bump `DEFAULT_GEMINI_MODEL` to `gemini-3.5-flash`, update the test mirror, refresh `.env.example`. The re-run passed 10/10 in 37.13 seconds — a 4× speedup, all explained by not retrying a dead endpoint.

**Class of bug:** vendor model IDs are versioned external state, not constants in your code. A `DEFAULT_MODEL` in `llm.py` is a contract with a remote API that can be revoked unilaterally. The same-day follow-up was migrating from the deprecated `google-generativeai` SDK to `google-genai`; the live eval re-ran green on the new SDK without changing the model id.

### BUG-011 — the test that only broke under concurrency

A parser test passed in isolation and failed intermittently in the full suite — roughly 60% flake rate. Either a partial-read `AssertionError` or a Windows `PermissionError [WinError 32]`.

The test helper `_render_to_text` wrote to a fixed path at the repo root: `tmp_test_render.log`. Two tests called the helper. In isolation: no contention. Inside the full suite: Windows file-watchers + antivirus + parallel pytest workers all hit the same path mid-write. Locks. Partial reads.

Fix: thread pytest's `tmp_path` fixture through every call so each invocation gets a unique OS-temp path. I verified with a two-thread stress harness — shared-path version failed 484/800; isolated version passed 800/800.

**Class of bug:** shared mutable state in tests is the same anti-pattern as shared mutable state in production. Flakiness is its visible failure mode. The test was *correct* in isolation; the bug only existed under concurrency. I now grep for repo-root writes in every test-helper review.

## The RAG contract that kept it honest

The Co-Pilot chat is the most opinionated part of the project. Three choices did most of the work:

**Hybrid retrieval over vector-only.** Vector retrieval alone misses exact matches like `Q15` or `0402`. rank-bm25 catches those. ChromaDB with `hnsw:space=cosine` catches semantic phrasing like "lifted leads" matching "open joints." Reciprocal Rank Fusion at k=60 combines both without tuning a weighted blend.

**Strict anti-hallucination grounding.** The `answer()` function refuses unless all four hold: (1) retrieval returned at least one hit, (2) the LLM emitted valid JSON, (3) the response's `sufficient` field is `true`, and (4) at least one cited chunk id is in the retrieved set. Hallucinated citations get dropped; the LLM is never called on an empty retrieval; the refusal text is a fixed constant, not LLM-generated.

**Eval as code, not vibe.** Ten frozen questions, one per failure-mode doc, in `tests/test_rag/eval_dataset.py`. Two layers: an offline citation-pattern test that proves the wiring with stubbed retrieval + stubbed LLM (runs in CI, zero cost); an env-gated live test that calls the real API and asserts ≥8/10 correct citations (runs on demand, one round-trip). The bar was later strengthened to ≥12/15 after fixing a terse-query retrieval miss (BUG-014). Both bars have been re-verified green.

## What I'd do differently

- **Migrate the vendor SDK earlier.** `google-generativeai` was already end-of-support when I adopted it. The cost of migrating later is rarely less than the cost of migrating now — especially when it's an SDK your critical path depends on.
- **Capture screenshots from CI, not by hand.** Fixed in Phase 4 slice 2 with a headless Playwright + Pillow pipeline; the six dashboard JPGs and a 748 KB demo gif regenerate from one command against a gitignored sample DB.
- **Bench real-data validation earlier.** The hard guardrail — real customer logs only on the work network — is correct, but it pushes real-data validation to the end. Earlier synthetic-but-realistic edge cases would have caught the parser BUG-007 that hardcoded `shift="A"` at the start of Phase 2 instead of mid-phase.

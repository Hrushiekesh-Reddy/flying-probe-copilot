## Revision 1 — 2026-06-16 — resolves Step 5 reviewer concerns

The Step 5 Plan Reviewer surfaced 7 BLOCKERs and 21 WARNINGs / MINORs against the original plan. This revision resolves the BLOCKERs in writing and folds the load-bearing WARNINGs in. The Executor MUST treat Revision 1 as overriding the original plan's locked decisions where they conflict; the original is preserved above for traceability. The Decision Gate (Step 6) surfaces all material choices below to the owner.

### R1-A — Pareto canonical SQL: drop "row-for-row match notebook Q2"  *(resolves BLOCKER 1 + 2)*
The notebook's Query 2 groups by `(record_type, failure_category)` and orders by `failures DESC` with NO tiebreak. Matching it row-for-row would require ParetoRow to carry both columns and use unstable ordering. Resolution: **`failure_pareto(by="record_type")` v1 groups by `record_type` ONLY** (collapses `failure_category` across rows) and uses tiebreak `key ASC` purely as an analytics-layer determinism choice — NOT a notebook match. Document the difference in `failure_pareto`'s docstring and in DECISION_LOG. The plan's L7 / L10 / L15 stay as-written but the JUSTIFICATION text "matches notebook ORDER BY" is **revoked** — tiebreak is locked for determinism, not for notebook parity. SUCCESS-WHEN #10 wording corrected: "matches a Pareto-by-record-type aggregation as defined by `failure_pareto`'s canonical SQL form (NOT notebook Query 2 directly)." Adding `failure_category` to ParetoRow is deferred to a later slice.

### R1-B — Yield row ordering universally `group_key ASC`  *(resolves BLOCKER 3)*
Notebook Q1 orders by `board_profile_id` ASC; Q4 orders by `panels_tested DESC, operator_id`. To avoid two different ordering rules per group_by, **lock yield rows to `group_key ASC` for ALL group_by values.** This matches Q1 row-for-row (Y-01 still valid) and diverges from Q4 (no Y-test asserts Q4 ordering, so no test breaks). Document the Q4 divergence in `yield_over_time`'s docstring + DECISION_LOG.

### R1-C — `yield_pct` and `pct_of_total` always unrounded floats  *(resolves BLOCKER 4 + test-plan G-1)*
`YieldRow.yield_pct` and `ParetoRow.pct_of_total` / `cumulative_pct` are stored as raw `float`. No `ROUND` in the SQL. Caller rounds at presentation. Document. Notebook Q3/Q4/Q5/Q6 differ — analytics layer does NOT match their rounded output. DECISION_LOG entry.

### R1-D — Pareto fixtures: per-test inline DB via `_make_pareto_db` helper  *(resolves BLOCKER 5)*
A single `pareto_fixture_db` cannot satisfy P-01 (strictly decreasing) + P-02 (ties at top) + P-05 (≥11 groups) + P-07 (ties at cutoff) + P-08 (mixed NULL refdes) + P-09 (in-window + out-of-window) simultaneously. **Resolution:** replace `pareto_fixture_db` with a `_make_pareto_db(records: list[dict]) → duckdb.Connection` helper in `tests/test_analytics/conftest.py` that takes a deterministic spec and returns a fresh in-memory DB. Each P-* test calls it with its own spec. Plan step #2 conftest description updated below in this revision.

### R1-E — Empty-anchor short-circuit  *(resolves BLOCKER 6 + test-plan G-7 + G-8)*
Locked: `_resolve_anchor(con, as_of=None) → datetime | None`. Returns `None` when `MAX(test_runs.start_ts) IS NULL` (empty DB or no rows in window). **Public functions check the return: if `None`, return `[]` immediately without executing the main SELECT.** `failure_pareto` adds the same short-circuit on `total == 0` after the WHERE filter (catches the `by="refdes"`-with-all-NULL case). Add `_window.py` module-level docstring explaining the short-circuit. Add tests `test_yield_with_empty_db_returns_empty_list` (Y-02 — already exists), `test_pareto_with_zero_failures_returns_empty_list` (P-10 — already exists), and NEW `test_pareto_by_refdes_with_all_null_refdes_returns_empty_list`.

```sql
WITH grouped AS (
  SELECT <key_col> AS key, COUNT(*) AS count
  FROM failures f JOIN test_runs tr ON tr.test_run_id = f.test_run_id
  WHERE tr.start_ts >= ? AND tr.start_ts <= ? [AND f.target_refdes IS NOT NULL]
  GROUP BY <key_col>
),
totals AS (SELECT SUM(count) AS overall_total FROM grouped),
ranked AS (
  SELECT key, count,
         100.0 * count / (SELECT overall_total FROM totals) AS pct_of_total,
         100.0 * SUM(count) OVER (ORDER BY count DESC, key ASC ROWS UNBOUNDED PRECEDING)
              / (SELECT overall_total FROM totals) AS cumulative_pct
  FROM grouped
)
SELECT * FROM ranked ORDER BY count DESC, key ASC LIMIT ?
```
Implication: with `top_n=3` and ≥4 groups, `cumulative_pct` of the last returned row is < 100. P-03 asserts `cumulative_pct ≈ 100.0` only when `top_n >= distinct_groups` — test description tightened accordingly.

### Test 4 — The exit-criterion query: yield by board over last week

```powershell
uv run python -c @"
import duckdb
con = duckdb.connect('data/db/qa.duckdb')
sql = '''
WITH anchor_cte AS (SELECT MAX(start_ts) AS anchor FROM test_runs)
SELECT
  p.board_profile_id,
  COUNT(*)                                                AS total,
  SUM(CASE WHEN tr.btest_status = 0 THEN 1 ELSE 0 END)    AS passed,
  ROUND(100.0 * SUM(CASE WHEN tr.btest_status = 0 THEN 1 ELSE 0 END) / COUNT(*), 1) AS yield_pct
FROM test_runs tr
JOIN panels p ON p.panel_serial = tr.panel_serial
WHERE tr.start_ts >= (SELECT anchor - INTERVAL 7 DAY FROM anchor_cte)
GROUP BY p.board_profile_id
ORDER BY p.board_profile_id;
'''
for r in con.execute(sql).fetchall():
    print(r)
"@
```

**Expected:**
- One row: `('small', 10, <~9 or 10>, <yield_pct around 90-100>)`.
- With `--fault-rate=0.05` (default) on 10 panels you get 0-2 failures by binomial chance.
- A value of `0%` would be alarming — that means btest_status is not 0 for any panel.

## What to look for

- **Stack traces from the parser CLI:** any uncaught exception = bug. Capture the full stack.
- **Wrong exit codes** on Tests 6 / 7: 0/1/2 must match exactly.
- **Wrong counts in Test 5**: panels and test_runs must match the generator's `--count`. Measurements should scale linearly with profile size × count.
- **Empty `failures` for a non-zero fault rate**: investigate — could indicate the parser is dropping non-PASS measurements.
- **`runs` table not populated after a successful ingest**: manifest.json read failure.
- **DuckDB file in `git status`**: means `.gitignore` didn't catch it; flag immediately.
- **`uv run parser --help`** should show `--input`, `--db`, `--encoding` (default `auto`) — confirm the help text is accurate.

## Pass/fail criteria

**PASS:** Tests 1-7 all behave as expected. Tests 8-9 succeed with the medium profile and show reasonable distributions.

**FAIL:** Any unexpected exit code, any uncaught exception, any count off by more than the documented tolerance, any DuckDB file appearing in `git status`. Note exactly what you saw — copy/paste the stdout/stderr.

If you find a FAIL: it goes to `docs/logs/BUG_LOG.md` and gets surfaced via a fresh `/session-workflow` Step 1 brief in the next session (just like Phase 1a's BUG-002/003 sprint).

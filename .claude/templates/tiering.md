# Session tiering — choosing a task class

> Generated by hrk-agent-starter / stamp.ps1. Customize the worked examples for your stack.
>
> Read at the start of every session AFTER the Document step's session brief is written.
> The tier you pick determines which steps of the session workflow actually run.
> Wrong tier = wasted tokens (over-engineered) or wasted time (under-engineered).

---

## The four tiers

> Step numbers below assume the kit's default 12-step session workflow.
> Reference by name (Document, Plan, Execute, Triple Check, Documentation) if your project has renumbered.

| Tier | Steps that run | Typical signal |
|------|----------------|----------------|
| **Trivial**  | Document → Triple Check → Documentation | Typo, single-line config, doc edit, README tweak |
| **Small**    | Document → Plan → Decision Gate → Execute → Triple Check → Documentation | One file, ≤200 LOC, one module, one concept |
| **Medium**   | Add Explore (before Plan) | New feature, 2–4 files, cross-module, but mechanism is clear |
| **Large**    | Full session workflow (all steps) | New subsystem, schema change, anything irreversible or with broad blast radius |

Skip in Trivial/Small/Medium: Test-Case Plan, Verify Plan, Verify Exec, Manual QA, Feedback+Handoff — these come back in Large.

---

## Five-minute tier decision (run at start of session)

Ask yourself, in order:

1. **Is the change one file, ≤30 lines, no logic?** → Trivial.
2. **Is the change one module, one concept, with tests already obvious?** → Small.
3. **Will the change touch ≥3 files OR introduce a new abstraction OR cross a milestone boundary?** → Medium.
4. **Is the change to a schema, migration, parser core, anything where a bug ships data corruption?** → Large.
5. **Is the change RISKY for any reason (irreversible, public-facing, runs on real data)?** → Bump one tier up.

If you can't pick in 5 minutes, default to **Medium** — it's the highest-value tier per token spent.

| Task pattern | Tier | Why |
|--------------|------|-----|
| Fix a typo in CLAUDE.md | Trivial | Doc edit, no code |
| Add a new CLI flag with obvious wiring | Small | One file, obvious test shape |
| Add a new config / data profile | Small | Config-shaped, fixture-shaped |
| Build a new bounded module (single concept) | Medium | New abstraction, multiple files, but mechanism is clear |

| Build a core subsystem (parser, scheduler, retriever) | **Large** | Bugs here corrupt downstream consumers |

| Wire up an external API (LLM, payment, auth) | **Large** | New external dependency, secret handling, broad blast radius |

## What each tier skips, and why

### Trivial — skips most sub-agent steps
The parent can do the whole thing inline. Triple Check is still run — even a typo deserves a read-back before commit. Documentation is still run because everything goes through the same logging discipline.

### Small — adds Plan + Decision Gate + Execute
You skip Explore because the parent already knows the touch points. You skip Verify-Plan / Verify-Exec because the work is small enough that the parent's own Triple Check is what catches errors. The Execute sub-agent IS used so tool restrictions still apply.

### Medium — adds Explore
Explore returns when the parent can't enumerate touch points from memory. Verify-Plan is still skipped — the plan is large enough that the parent's own Triple Check read is what catches errors.

### Large — full loop
Test-Case Plan + Verify-Plan + Verify-Exec + Manual QA come back. The token cost is justified by the blast radius — a bug in a core subsystem corrupts every downstream consumer.

## Mid-session escalation (what if the tier was wrong?)

If you started Trivial/Small/Medium and discover the work actually needs a higher tier (e.g. the "small" CLI flag turns out to require a schema column), DO NOT keep going at the current tier. Escalate explicitly:

### Escalation protocol

1. **STOP the current sub-agent immediately.** Do not let it keep working with the wrong tier's guardrails.
2. **Log the escalation** in the session brief under a `### Tier escalation` section:
   ```
   ### Tier escalation — [timestamp]
   Original tier: Small
   New tier:      Medium
   Trigger:       Found that --seed flag requires a new column in the run_metadata table
   Cost so far:   ~3k tokens, no commits, no edits to gated files
   ```
3. **Reset the session brief.** The OUT-OF-SCOPE block and SUCCESS-WHEN criteria may now be wrong — rewrite them.
4. **Restart from the appropriate step of the new tier.**
   - Trivial → Small: restart at Plan, because you now need one.
   - Small → Medium: restart at Explore, because you need touch-point mapping.
   - Medium → Large: restart at Test-Case Plan / Verify Plan, because the blast radius now justifies adversarial review.
5. **Tell the owner.** Escalations are informative — owner may decide to defer the work to a separate session instead of expanding the current one.

### Anti-pattern (do not do this)

- "I started this as Small but I'll just add an Explore step and keep going." → No. The current sub-agent has stale context. Stop, reset, restart cleanly.
- "I'll quietly add the schema column inline since the parent will catch it." → No. Approval-gated files require an explicit pause regardless of tier.
- Treating escalation as failure → it's not. Catching a wrong tier 5k tokens in is a feature; catching it 50k tokens in is a bug.

---

## Anti-tier (when to DECLINE to start)

If you can't pick a tier even after the five-minute decision, the request is under-specified. STOP at the Document step. Ask the owner to clarify. Bad tier choice on a vague request burns 10x the tokens of a clarifying question.
